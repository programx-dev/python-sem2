# Отчет по лабораторной работе №1

**Тема:** Источники задач и контракты
**Предметная область:** Платформа обработки задач

## 1. Цель работы

Целью работы является реализация подсистемы приема задач в платформе обработки задач. Основная сложность заключается в необходимости поддержки множества разнородных источников данных (файлы, генераторы, внешние API), которые не должны быть связаны жесткой иерархией наследования, но обязаны реализовывать единый поведенческий контракт. Система должна позволять добавлять новые типы источников без изменения существующего кода и иметь аннотации всех публичных интерфейсов.

## 2. Архитектурные решения

В ходе работы были приняты следующие ключевые архитектурные решения:

### 2.1. Контрактное программирование

Вместо наследования ООП использована **структурная типизация** через `typing.Protocol`.

* **Решение:** Описан протокол `TaskSource`, определяющий метод `get_tasks() -> Iterable[Task]`.
* **Преимущество:** Это позволило полностью отвязать реализации источников от ядра системы. Любой объект, обладающий нужным методом, автоматически считается валидным источником.

### 2.2. Паттерн "Репозиторий"

Для реализации расширяемости без модификации кода применен паттерн регистрации через декораторы.

* **Решение:** Создан модуль `repository.py`, содержащий словарь `REGISTRY` и декоратор `@register_source`.
* **Преимущество:** Новые источники регистрируются в системе автоматически при импорте модуля.

### 2.3. Ядро системы и агрегация данных

Класс `InboxTasks` выступает в роли агрегатора данных.

* **Проверка:** Валидация контракта источников происходит в методе `__init__` через `isinstance(source, TaskSource)`. Это предотвращает работу системы с некорректными объектами.
* **Ленивые вычисления:** Использование `Iterable` и ключевого слова `yield` гарантирует, что задачи обрабатываются потоково, не загружая все задачи в ОЗУ, это дает преимущество при больших объемах данных.

### 2.4. Аннотации

В аннотациях интерфейсов и контрактов было принято решение использовать наиболее абстрактный тип `Iterable` вместо конкретных структур данных (`list`, `tuple`). Использование `Iterable` накладывает минимально необходимые ограничения на источник данных: объект должен поддерживать только перебор в цикле.

* **Преимущество:** Это позволяет источникам использовать генераторы (ленивые вычисления). Данные не загружаются в оперативную память целиком, а обрабатываются по одной задаче за раз, что критично при работе с большими JSONL-файлами или бесконечными потоками задач.

## 3. Особенности реализации и допущения

### 3.1. Преобразование данных

Принято допущение, что каждый источник самостоятельно знает структуру своих исходных данных и преобразует их в унифицированный объект `Task`. Это освобождает ядро системы от знания деталей реализации конкретных форматов.

### 3.2. Обработка исключений

Для обеспечения надежности была реализована обработка исключений:

* **Ошибки:** Источники обрабатывают системные ошибки (`FileNotFoundError`, `PermissionError`, `JSONDecodeError`), так и неверный ввод данных пользователя с консоли или из файла.

## 4. Тестирование

Система покрыта автоматическими тестами на **96%**. Тестирование проводилось с использованием библиотек `pytest` и `pytest-cov`.

**Что покрыто тестами:**

* **Unit-тесты контрактов:** Проверка корректности работы `isinstance` для протоколов.
* **Тесты сущностей:** Проверка неизменяемости объектов `Task`. Тестирование всех источников задач.
* **Mock-тестирование:** Использование простых источников для проверки логики агрегации в `InboxTasks` с использование механизма фикстур.
* **Тесты CLI:** Использование `CliRunner` для проверки корректности работы всей программы от ввода команды до вывода результата в консоль.
* **Тестирование негативных сценариев:** Проверка реакции системы на невалидные JSON-файлы и некорректные типы данных.

## 5. Доступные источники задач

Система поддерживает три типа источников, каждый из которых предназначен для определенных сценариев:

* **`generator`** — Программный источник. Генерирует указанное количество задач с полезной нагрузкой в виде случайных чисел. Удобен для нагрузочного тестирования и проверки логики ядра без внешних данных.
* **`jsonl`** — Файловый источник. Читает задачи из файлов формата JSON Lines. Поддерживает валидацию структуры JSON и информирование об ошибках с указанием номера строки.
* **`api-stub`** — Имитационный источник (заглушка). Моделирует работу внешнего API, выдавая случайный набор задач из заранее определенного списка констант. Поддерживает настройку диапазона количества выдаваемых задач.

## 6. Инструкции по запуску в Docker

Проект можно запускать с помощью Docker. Docker-образ оптимизирован: используется многоэтапная сборка и кэширование слоев, по этой причине команды сборки расположены в таком порядке. Для ускорения скачивания зависимостей используется `uv` вместо стандратного `pip`.

### 6.1. Сборка образа

Находясь в корневой папке лаборатоной работы (т.е. в папке `lab-01/`):

```bash
docker build -t lab-01 .
```

*Для всех последуючих команд используется флайг --rm для автоматического удаления контейнерра после его остановки.*

Для удобства проверки внутрь контейнера прeдуставновлен фалй с тестовыми данными, находящийся по пути `./data/tasks.jsonl` внутри контейнера.

### 6.2. Варианты запуска программы

**Пример 1: Комбинированный запуск всех источников**  
(Чтение из файла, генерация 6 задач и активация API со случайным количеством задач от 3 до 7):

```bash
docker run --rm lab-01 read --jsonl ./data/tasks.jsonl --gen 6 --api --api-min 3 --api-max 7
```

**Пример 2: Чтение из генератора (5 задач):**

```bash
docker run --rm lab-01 read --gen 5
```

**Пример 3: Запуск только API заглушки с фиксированным количеством задач:**

```bash
docker run --rm lab-01 read --api --api-min 5 --api-max 5
```

**Пример 4: Настройка диапазона API (от 2 до 4 задач):**

```bash
docker run --rm lab-01 read --api --api-min 2 --api-max 4
```

**Пример 5: Чтение из файла:**

```bash
docker run --rm lab-01 read --jsonl data/tasks.jsonl
```

**Пример 6: Просмотр доступных типов источников:**

```bash
docker run --rm lab-01 sources
```

### 6.3. Запуск тестов и проверка покрытия внутри Docker

**Запуск всех тестов:**

```bash
docker run --rm --entrypoint pytest lab-01
```

**Запуск тестов с отчетом по покрытию:**

```bash
docker run --rm --entrypoint pytest lab-01 --cov=src
```
