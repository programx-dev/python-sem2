# Отчет по лабораторной работе №3

**Тема:** Очередь задач: итераторы и генераторы  
**Предметная область:** Платформа обработки задач

## 1. Цель работы

Целью работы является освоение механизмов реализации пользовательских коллекций и ленивой обработки данных. Основное внимание уделено проектированию системы потоковой фильтрации задач с использованием протоколов итерации (`Iterable`, `Iterator`) и генераторов, что позволяет минимизировать потребление оперативной памяти при работе с большими объемами данных.

## 2. Архитектурные решения

В ходе реализации подсистемы хранения и фильтрации были применены следующие подходы:

### 2.1. Расширение InboxTasks

Компонент `InboxTasks` был модернизирован для поддержки протокола итерации. Реализация магического метода `__iter__` позволяет использовать объект как итерируемый. При каждом вызове итератора создается новый генератор через метод `fetch_all()`, что обеспечивает повторный обход данных из внешних источников без их кэширования в памяти.

### 2.2. Ленивая очередь задач

Реализован класс `TaskQueue`, представляющий собой декоратор над итерируемым источником задач.
*   **Отложенные вычисления:** Очередь не хранит задачи физически, а лишь ссылку на итерируемый объект `Iterable[Task]` и список фильтров (предикатов), которым каждая задача должна удовлетворять.
*   **Метод `filter_by`:** Реализует такой подход, что каждый вызов возвращает новый экземпляр `TaskQueue`, дополненный новым условием фильтрации. Это позволяет строить сложные цепочки фильтров.

### 2.3. Специализированный итератор TaskQueueIterator

Для управления процессов обхода задач из источника и проверки их соответсвия заданному набору фильтров приняты следующие решения:
*   **Метод `__next__`:** Содержит основную логику фильтрации. Перебор элементов из базового источника выполняется до тех пор, пока не будет найдена задача, удовлетворяющая всем установленным фильтрам.
*   **Обработка `StopIteration`:** Итератор корректно завершает работу, когда внутренний источник данных исчерпан, выбрасывая стандартное исключение `StopIteration`, которое распознается циклом `for`.

## 3. Особенности реализации

### 3.1. Эффективность и потоковая обработка

Главным преимущетсвом системы является постоянное потребление памяти $O(1)$. Задачи считываются из источников, валидируются дескрипторами и фильтруются по одной. Это предотвращает переполнение памяти даже при обработке бесконечных или больших потоков данных.

## 5. Инструкции по запуску в Docker

Сборка и запуск контейнера аналогичны предыдущим работам, с добавлением новых флагов фильтрации.

### 5.1. Сборка образа

Находясь в корневой папке лабораторной работы №3:

```bash
docker build -t lab-03 .
```

### 5.2. Варианты запуска программы

**Пример 1: Сложная фильтрация (JSONL + API + Генератор):**
```bash
docker run --rm lab-03 read --jsonl ./data/tasks.jsonl --api --gen 10 --min-priority 5 --status new
```

**Пример 2: Фильтрация по нескольким статусам (только 'done' и 'in_progress'):**
```bash
docker run --rm lab-03 read --api --api-min 10 --api-max 30 --status done --status in_progress
```

**Пример 3: Использование очереди без фильтров (вывод всех задач):**
```bash
docker run --rm lab-03 read --gen 5
```

**Пример 4: Просмотр доступных плагинов:**
```bash
docker run --rm lab-03 sources
```

### 5.3. Запуск тестов и проверка покрытия

Для подтверждения корректности работы итераторов и отсутствия утечек памяти:

```bash
docker run --rm --entrypoint pytest lab-03
```
