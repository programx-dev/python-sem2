# Отчет по лабораторной работе №2

**Тема:** Модель задачи: дескрипторы и @property  
**Предметная область:** Платформа обработки задач

## 1. Цель работы

Целью работы является проектирование и реализация модели задачи с использованием механизмов Python: дескрипторов данных (data descriptors), дескрипторов не-данных (non-data descriptors) и `@property`. Основная задача - обеспечить защиту инвариантов доменной модели.

## 2. Архитектурные решения

В ходе реализации модели `Task` были применены следующие архитектурные подходы:

### 2.1. Использование дескрипторов для валидации

Вместо ручной проверки типов в методе `__init__`, валидация вынесена в специализированные классы-дескрипторы, наследующиеся от `BaseDataDescriptor` либо в `@property`.

* **NotEmptyStrDescriptor:** Гарантирует, что описание задачи не является пустой строкой и автоматически удаляет лишние пробелы.
* **PriorityDescriptor:** Реализует проверку бизнес-правила: приоритет должен быть целым числом в отрезке от 0 до 100.
* **EnumValidator:** Обеспечивает строгую типизацию статуса задачи, проверяя принадлежность значения к перечислению `TaskStatus`.
* **DatetimeDescriptor:** Универсальный инструмент для работы с датами. Поддерживает как объекты `datetime`, так и строки (с автоматическим приведением). Содержит механизм `read_only`, предотвращающий изменение даты создания после инициализации.

### 2.2. Разграничение Data и Non-Data Descriptors

Для демонстрации различных уровней доступа реализованы два типа дескрипторов:

* **Data Descriptors**: имеют метод `__set__`, полностью контролируют доступ к атрибуту и имеют приоритет над словарем экземпляра `__dict__`.
* **Non-Data Descriptors** (`MetadataDescriptor`): имеют только метод `__get__`. Используются для предоставления динамической информации об объекте (например, строкового представления типа и ID), не мешая при этом обычным операциям записи в `__dict__` и перекрытия видимости этого дескриптора.

### 2.3. Механизм `@property` для связанных инвариантов

Для атрибутов, чья валидация зависит от состояния других полей, использован декоратор `@property`:

* **Свойство `deadline`:** Содержит сложную логику проверки. Срок выполнения задачи (`deadline`) не может быть установлен раньше даты создания (`created_at`). Поскольку `created_at` сам является динамическим атрибутом, проверка вынесена в `.setter`.
* **Вычисляемые свойства (`is_overdue`, `remaining_time`):** Позволяют получать производные данные на основе текущего времени и установленного дедлайна, не сохраняя их в памяти, что исключает рассинхронизацию данных.

## 3. Особенности реализации и инварианты

### 3.1. Защита от некорректных состояний

Система предотвращает перевод объекта в невалидное состояние на нескольких уровнях:

1. **При инициализации:** Если в словаре (например, из JSONL) приходят некорректные типы, дескрипторы выбрасывают `TaskException`.
2. **При модификации:** Попытка изменить дату создания или установить отрицательный приоритет у существующего объекта также будет остановлена.
3. **Синхронизация времени:** Логика дедлайна гарантирует временную последовательность событий задачи.

### 3.2. Обработка исключений

Создан собственный набор специализированных исключений, как при попытке нарушить инварианты, так и при попытке считать данные, неудовлетворяющие источникам (например чтенеие из .jsonl файла).

## 4. Тестирование

Модель `Task` покрыта тестами, проверяющими:

* **Корректную инициализацию:** Создание объекта с валидными данными.
* **Валидацию границ:** Приоритеты < 0 или > 100.
* **Типизацию:** Передача строк вместо чисел или объектов `Enum`.
* **Иммутабельность:** Попытка изменить поля с флагом `read_only`.
* **Логику дат:** Проверка запрета на установку дедлайна в прошлом относительно момента создания.

## 5. Инструкции по запуску в Docker

Проект можно запускать с помощью Docker. Docker-образ оптимизирован: используется многоэтапная сборка и кэширование слоев, по этой причине команды сборки расположены в таком порядке. Для ускорения скачивания зависимостей используется `uv` вместо стандратного `pip`.

### 5.1. Сборка образа

Находясь в корневой папке лаборатоной работы (т.е. в папке `lab-02/`):

```bash
cd lab-02
```

```bash
docker build -t lab-02 .
```

*Для всех последуючих команд используется флайг --rm для автоматического удаления контейнерра после его остановки.*

Для удобства проверки внутрь контейнера прeдуставновлен фалй с тестовыми данными, находящийся по пути `./data/tasks.jsonl` внутри контейнера.

### 5.2. Варианты запуска программы

**Пример 1: Комбинированный запуск всех источников**  
(Чтение из файла, генерация 6 задач и активация API со случайным количеством задач от 3 до 7):

```bash
docker run --rm lab-02 read --jsonl ./data/tasks.jsonl --gen 6 --api --api-min 3 --api-max 7
```

**Пример 2: Чтение из генератора (5 задач):**

```bash
docker run --rm lab-02 read --gen 5
```

**Пример 3: Запуск только API заглушки с фиксированным количеством задач:**

```bash
docker run --rm lab-02 read --api --api-min 5 --api-max 5
```

**Пример 4: Настройка диапазона API (от 2 до 4 задач):**

```bash
docker run --rm lab-02 read --api --api-min 2 --api-max 4
```

**Пример 5: Чтение из файла:**

```bash
docker run --rm lab-02 read --jsonl data/tasks.jsonl
```

**Пример 6: Просмотр доступных типов источников:**

```bash
docker run --rm lab-02 sources
```

### 5.3. Запуск тестов и проверка покрытия внутри Docker

**Запуск всех тестов:**

```bash
docker run --rm --entrypoint pytest lab-02
```
